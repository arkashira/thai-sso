## Diagnosis  

The current `logout` implementation has three critical problems that cause runtime failures and can leave stale session data:

| # | Problem | Why it breaks |
|---|----------|---------------|
| 1 | **Mutating `sessions` while iterating** – `for sid, session in sessions.items(): … del sessions[session_id]` | Python raises `RuntimeError: dictionary changed size during iteration`. |
| 2 | **Accessing a deleted entry** – after `del sessions[session_id]` the code still reads `sessions[session_id].user.id` | Raises `KeyError` and may corrupt the logout logic. |
| 3 | **No thread‑safety** – `sessions` is a plain global dict that can be read/written from multiple request threads. | Leads to race conditions, lost updates, or corrupted state under load. |

The fix must:

* Capture any needed data **before** the deletion.
* Iterate over a **stable snapshot** of the dict (e.g., `list(sessions.items())` or `list(sessions.keys())`).
* Delete entries **without** later referencing them.
* Guard all reads/writes with a lock so the mapping is safe in a concurrent environment.
