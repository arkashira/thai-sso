This correctness issue must be fixed before the change can be accepted.

- The loop `for sid, session in sessions.items():` deletes entries from `sessions` inside the iteration, causing a `RuntimeError: dictionary changed size during iteration`.
- After `del sessions[session_id]`, the code still accesses `sessions[session_id].user.id` inside the loop, which will raise a `KeyError`.
- The function should first capture the user ID, then safely iterate over a copy of the session keys (or build a list of sessions to delete) and delete them without referencing the already‑removed entry.
- Ensure that the global `sessions` mapping is correctly defined and thread‑safe if used in a concurrent environment.

## 1. Acceptance Criteria  
1. **Secure Authentication** – The `/login` endpoint must accept a Thai SSO token, validate it via the Thai SSO provider, and establish a session only if the token is valid.  
2. **PDPA‑Compliant Data Handling** – All personal data (e.g., name, email, national ID) retrieved from the SSO provider must be stored **only** in encrypted form in the local database and never logged in plaintext.  
3. **Session Management** – Upon successful login, a JWT (or session cookie) is issued with a 30‑minute inactivity timeout and a 24‑hour absolute expiry.  
4. **Logout** – The `/logout` endpoint must terminate the current session and invalidate all active tokens for that user across all devices.  
5. **Error Handling** – Invalid or expired tokens must return a `401 Unauthorized` with a generic error message (no sensitive data).  
6. **Audit Trail** – All login and logout events must be recorded in an audit log with timestamp, user ID, and IP address, but without storing the raw SSO token.  
7. **Rate Limiting** – The login endpoint must enforce a limit of 5 attempts per IP per 15 minutes to mitigate brute‑force attacks.
